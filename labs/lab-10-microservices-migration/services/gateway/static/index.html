Lab 10: Microservices Migration

This lab splits the chat system into a gateway, a write service, and a history service. The gateway handles websocket ingress, rate limiting, and trace propagation; the write service persists and publishes; the history service serves room history as a separate read path.

Why split the monolith
A mature chat platform eventually needs separate deployable units. Splitting ingress, writes, and reads lets each concern scale and fail independently, while the gateway keeps the client contract stable.
Tracing in this lab
Every message is stamped with a trace id at the gateway and forwarded through the write service into the database. The history service also reports the latest trace it saw so you can follow the request path across services.
Rate limiting
The gateway enforces per-user token buckets. Bursty users will hit 429 responses while normal chat flows continue for other users. This highlights gateway-owned controls in a microservices design.
What to observe
Watch gateway request latency, rate limit behavior, the write service status, and the history service query path. A successful migration keeps the client UI simple while moving complexity behind service boundaries.
