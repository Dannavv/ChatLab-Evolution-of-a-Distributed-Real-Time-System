Lab 09: Message Security and Trust

Secure the chat mesh with encrypted envelopes, HMAC signatures, replay protection, key rotation, and rate controls. The UI lets you send normal messages, inject tampered signatures, and rotate keys while watching both nodes converge on the same security state.

What this lab teaches
This lab focuses on trust boundaries inside a chat system: who can send, how messages are signed, how they are encrypted, and how the system reacts when signatures fail or replay attempts appear.
Security mechanics
Each message is encrypted with AES-GCM and signed with an HMAC derived from the room key version. Nodes reject invalid signatures, drop replayed IDs, and rotate keys without taking the chat down.
Abuse controls
Each user has a local token bucket. When one user sends too fast, the API stops accepting messages before the mesh gets flooded. The UI exposes the current rejection counters so you can see throttling and recovery.
What to observe
Watch the current key version, accepted versus rejected counts, and the signature failure metrics. Rotate keys from the UI and confirm both nodes converge on the same version through Redis.
