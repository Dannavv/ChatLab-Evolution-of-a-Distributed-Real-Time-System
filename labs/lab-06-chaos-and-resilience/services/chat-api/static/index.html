Lab 06: Chaos and Resilience

Inject faults into the chat ingress path, trigger retries and breaker behavior, then verify the worker pipeline remains idempotent and controlled under failure pressure.

What this lab proves
Resilience is not just adding retries. This lab demonstrates bounded retries, circuit-breaker guardrails, and idempotent worker execution to prevent retry storms and duplicate delivery.
Failure controls in this UI
Set API chaos drop-rate and artificial delay. As failure probability rises, enqueue retries increase. If failures stay consecutive, the circuit breaker opens and rejects requests briefly.
Worker-side resilience
Workers use queue-inflight handoff, retry budget, dead-letter queue, and idempotency keys. This avoids duplicate writes and supports predictable recovery after transient errors.
Suggested test loop
Start with no chaos and send bursts. Then set drop-rate to 0.4 and delay to 120ms. Watch breaker state and queue depth, then reset chaos to 0 to observe recovery.
