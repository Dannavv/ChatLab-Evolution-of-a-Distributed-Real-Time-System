Lab 04: Scalable Monolith

Async message handling for a chat system that still runs as one deployment unit. Messages are accepted quickly, buffered in memory, and processed by background workers so the hot path stays responsive under bursty load.

What this lab changes
Lab 03 solved cross-node delivery. Lab 04 keeps the system monolithic, but shifts the expensive work out of the request path. Incoming chat messages enter a bounded queue, workers drain the queue in the background, and the live dashboard shows how many messages are waiting versus actively processed.
Why the queue matters
A monolith can still scale if the server separates admission from processing. The queue lets WebSocket readers stay fast while worker goroutines handle downstream work. When the queue fills, the lab drops messages and increments metrics so you can see the cost of insufficient capacity.
What to watch
Watch queue depth, busy workers, active connections, and message latency. A healthy server should keep the queue shallow, process messages steadily, and avoid sustained drops when you burst traffic from the UI or k6.
Run and benchmark
Start the stack with Docker Compose, then open the UI and send bursts of messages. The same endpoints are available for load testing through the shared k6 harness.
