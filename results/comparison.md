## Progression Summary

| Lab | Scaling Inflection | Traceability | Observability |
|---|---|---|---|
| lab-01-monolith-baseline | Shared in-memory state is fast but non-durable and non-scalable. | trace_id and message_id are generated by the client and echoed by the server with server_receive_ts and server_broadcast_ts when available. | Prometheus metrics plus per-run benchmark artifacts. |
| lab-02-persistence-layer | Durability improves correctness but adds a measurable synchronous write tax. | trace_id and message_id remain stable through ingest, persistence, and echo paths. | Prometheus metrics, DB latency histogram, and benchmark summaries. |
| lab-03-redis-pubsub | Fan-out scales better than a mutex-bound monolith, but consistency becomes eventual. | trace_id and message_id survive Redis publish and subscriber rebroadcast. | Prometheus metrics plus duplicate detection in benchmark summaries. |
| lab-04-scalable-monolith | Queueing delays replace lock contention as the dominant bottleneck. | trace_id and message_id survive enqueue, worker execution, and rebroadcast. | Prometheus metrics, memory tracking, and benchmark summaries. |
| lab-05-cloud-native-chat-infrastructure | Burst tolerance improves, but end-to-end semantics are now explicitly asynchronous. | trace_id and message_id propagate through API ingress, queue handoff, worker processing, and echo. | Prometheus metrics across API and worker components. |
| lab-06-chaos-and-resilience | Resilience protects throughput during failure, but it must be measured as an explicit trade-off. | trace_id and message_id survive retries, dead-letter handling, and final broadcast. | Prometheus metrics plus benchmark summaries that surface delivery loss. |
| lab-07-real-time-presence-and-delivery | Presence turns state freshness into a first-class scaling problem separate from message durability. | trace_id and message_id identify message, presence, and delivery events across fan-out. | Prometheus metrics plus benchmark visibility into connection count and throughput. |
| lab-08-global-multi-region | Geography forces a formal consistency trade-off and a formal routing policy. | trace_id and message_id survive regional ingest, bridge forwarding, and remote rebroadcast. | Prometheus metrics on regional nodes and the bridge service. |
| lab-09-message-security | Confidentiality and integrity change the system budget even when topology stays similar. | trace_id and message_id survive signing, verification, decrypt, and broadcast. | Prometheus metrics plus security-specific failure paths surfaced in the service. |
| lab-10-microservices-migration | The architecture becomes operationally composable, but routing, tracing, and cost must be managed as first-class concerns. | trace_id and message_id propagate through gateway, message service, history service, and storage. | Prometheus metrics plus trace IDs propagated through HTTP headers and messages. |
